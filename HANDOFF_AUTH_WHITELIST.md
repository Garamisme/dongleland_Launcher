# 동글랜드 런처 — 인증 / 화이트리스트 아키텍처 인수인계

> 작성 목적: "무료 프로그램 라이선스" 논의가 최종적으로 **정품 인증 + 서버 측 화이트리스트**
> 구조로 수렴한 과정과 결과를 정리한다. 이후 작업자는 폐기된 방향을 다시 파지 않도록 한다.

---

## 0. 한 줄 요약

**런처는 검증하지 않는다. MS 계정으로 로그인해 정품 UUID만 얻고 게임을 실행한다.
접속 통제(진짜 방어선)는 마인크래프트 서버의 `whitelist.json`이 담당한다.
멤버 데이터는 운영자 측 SQLite에서 관리하고, 거기서 `whitelist.json`을 export 한다.**

---

## 1. 최종 아키텍처

```
[런처]  MS 계정 OAuth 로그인 → 정품 UUID 획득 → 올바른 JVM 인자로 마크 실행
                                        (※ 접속 가부를 판단하지 않음. 실행만.)
             │
             ▼
[SQLite DB]  members 테이블에 UUID + 부가데이터 관리 (운영자/서버 측)
             │  export
             ▼
[whitelist.json]  마크 서버가 접속 시 UUID 대조 → 허용 / 거부
                  ◀── 실제 접속 통제는 여기. 100% 서버 측(server-side).
```

핵심 원칙: **클라이언트(런처)를 신뢰하지 않는다.** 런처는 크랙 가능하므로
어떤 접속 권한 판단도 런처에 두지 않는다. 판단은 전적으로 마크 서버가 한다.

---

## 2. 왜 이 구조인가 (결정 이력)

논의는 여러 방향을 검토한 뒤 아래처럼 좁혀졌다. **폐기된 방향을 다시 검토하지 말 것.**

| 검토안 | 결론 | 이유 |
|---|---|---|
| Ed25519 라이선스 키 (오프라인+온라인 하이브리드) | **폐기** | 정품 인증이 식별·검증·통제를 이미 다 해줌. 같은 일 두 번. |
| 언어 교체(Rust/Go 등)로 크랙 방지 | **폐기** | 클라이언트 검증은 언어 불문 원리적으로 우회 가능. 무의미. |
| MSIX + 윈도우 스토어 배포 | **보류** | 스토어는 마크 IP/심사/속도 문제. MSIX 포맷도 지금은 불필요. |
| exe 코드 사인 (OV/EV 인증서) | **보류** | 연간 비용 + 법인 요구. 소규모 커뮤니티엔 SmartScreen 안내로 충분. |
| Discord 봇 발급 | **폐기** | Discord 사용 안 함. |
| **MS 계정 정품 인증 + 서버 whitelist** | **채택** | 무료·표준·견고. 별도 키 시스템 불필요. |

### 핵심 통찰
- "무료 프로그램 라이선스"의 진짜 목적은 과금이 아니라 **식별 + 접속 통제**다.
- 정품 인증(MS OAuth) 하나가 그 목적을 전부 충족한다:
  - **식별** — 계정 UUID로 사용자 확정
  - **정품 검증** — 무단 복제 계정 차단
  - **통제** — 서버 화이트리스트에 UUID 등록/제거
- 따라서 별도 라이선스 키 발급/서명/취소 시스템은 **불필요**.

## 3. 비용 / 리스크 정리

| 요소 | 비용 성격 | 현재 판단 |
|---|---|---|
| exe 코드 사인 | **실제 금전** (인증서 연간, 법인 검증 요구) | 보류. 규모/신뢰 필요 시 EV 추가 |
| MS 계정 연동 (Azure 앱 등록) | 무료 | 채택. 단 정책·유지보수 부담 존재 |
| 게임 실행 로직 | 구현 시간만 | 기존 런처 로직의 연장 |
| SmartScreen 경고 | 없음(안내로 대응) | "추가 정보 → 실행" 안내로 커버 |

> ⚠️ MS 인증 플로우는 변경될 수 있음. Mojang 계정은 이미 폐지, MS 계정 OAuth만 유효.
> 서드파티 런처용 OAuth는 회색지대이므로 Minecraft EULA/브랜드 정책 준수 필요.

## 4. 데이터베이스 (SQLite)

**source of truth = 이 DB.** `whitelist.json`은 여기서 export 되는 파생물이다.

### 저장소 선택 근거
- 지금은 마크 내장 `whitelist.json`만으로도 충분하나, **부가 데이터 확장 여지**를 위해 SQLite 채택.
- 서버 프로세스 불필요(파일 1개). 나중에 Postgres 이전 시 SQL 거의 그대로 이식됨.
- **온라인 DB는 "여러 관리자가 원격 실시간 수정"이 필요할 때만** 정당화됨. 현재는 불필요.

### 테이블 구성
- `members` — 핵심. 사람 1명 = 1행. UUID + 닉네임 + status(active/suspended/banned/pending).
  부가 컬럼(role, joined_at, last_login_at, login_count, note, metadata)은 NULL 허용으로 미리 열어둠.
- `login_events` — 로그인/실행 이력(통계·감사용). 지금은 비워둬도 무방.
- `v_whitelist` (뷰) — **핵심 편의 장치.** 하이픈 없이 저장된 UUID를 마크 형식
  (`8-4-4-4-12` + name)으로 자동 변환하고, `status='active' AND is_whitelisted=1`만 필터링.

### UUID 저장 규칙
- DB 저장: **하이픈 없는 소문자 32자** (Mojang API 형식)
- export 시: `v_whitelist` 뷰가 하이픈 형식으로 변환 (whitelist.json 요구 형식)

### 운영 흐름
```
멤버 추가/변경  →  members UPDATE
       ↓
v_whitelist 뷰 SELECT
       ↓
whitelist.json export  →  마크 서버 반영
```
- 멤버 정지 = `status='suspended'`로 변경. **삭제하지 않음** → 다음 export 때 자동 제외되며 이력·부가데이터 보존.

### 설계 판단 근거
- **삭제 대신 status 변경** — 정지 사유·가입 이력 추적 위해 데이터 보존.
- **whitelist.json이 아니라 DB가 원천** — 부가데이터+접속권한 단일 관리.
- **IP는 원문 대신 해시** — 통계 필요 시에도 개인정보 원문 저장 회피.
- **DB는 런처가 아니라 운영자/서버 측에서 관리** — 런처에 DB 물리면 "클라이언트 불신" 원칙 위배.

관련 파일: `schema.sql`

## 6. 다음 작업자 주의사항

1. **라이선스 키 시스템을 다시 만들지 말 것.** 정품 인증으로 대체되어 폐기됨.
2. **접속 판단을 런처에 넣지 말 것.** 크랙되면 무의미. 판단은 마크 서버가 전담.
3. **코드 사인은 지금 도입하지 말 것.** 비용 대비 실익 없음. 규모 커지면 그때 EV.
4. **DB는 운영자 측.** 런처 배포물에 DB나 화이트리스트 원본을 포함하지 말 것.
