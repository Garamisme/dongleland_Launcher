- 문서 버전: 1.1 (Draft) — **서드파티 전제 반영 개정**
- 작성일: 2026-07-10
- 대상 릴리즈: v3.1.0 (필요 시 3.1.x / 3.2.0 분할 — §7 참조)

## 0. 핵심 전제 (v1.1 개정 사유)

**동글랜드 런처는 서드파티 앱이다. 동글랜드 서버는 제3자가 운영하며, 서버 측에 코드를 배포할 수 없다.**

따라서 모든 기능은 다음 두 자산만으로 성립해야 한다:
1. **클라이언트 측** — 런처 + 자체 Fabric 모드 (유저 PC에서 실행)
2. **자체 백엔드** — 게임 서버와 무관하게 독립 운영

이는 루나/Feather 클라이언트와 동일한 모델이다 — 그들 역시 하이픽셀 등 어떤 서버에도 서버 측 코드 없이 친구·파티·HUD를 제공한다. 서버 운영진과의 협업(플러그인 설치 등)은 **의존해서는 안 되는 선택적 확장**으로만 취급한다 (§9-D4).

## 1. 개요

v3.0(풀 게임 클라이언트 전환: MS OAuth 로그인, mll 직접 설치·실행, quickPlay 자동접속)을 기반으로, 트랙 A "런처 고도화"를 구현한다. 범위는 ① 디자인 리뉴얼(테마 기반 구조), ② 자체 백엔드 최소 구축, ③ 동글랜드 코어 모드(자체 Fabric 모드) 탑재, ④ 소셜 기능(친구/파티/동반접속)이다.

트랙 A의 완료 기준: **"동글랜드 유저가 이 런처로 모드·소셜을 쓰는 게 당연한 상태"**. 트랙 B(타이쿤 클라이언트 플랫폼화)의 토대가 되므로, 코어 모드와 백엔드는 동글랜드 전용 하드코딩을 최소화한다.

## 3. 범위

### 포함 (In Scope)
- F1 디자인 리뉴얼 — 시안 HTML 기반 프론트 교체 + CSS 변수 체계화
- F2 자체 백엔드 최소 구축 — 런처 API 서버 + 회원/소셜 DB
- F3 동글랜드 코어 모드 — 클라이언트 사이드 Fabric 모드, 내장 배포, 관찰 기반 데이터 계층
- F4 소셜 기능 — 친구, 파티, 접속 상태, 동반 접속 (전부 자체 백엔드 기반)
- F5 소식 카드 — 홈 화면 공지/이벤트 슬롯

### 제외 (Out of Scope)
- **서버 측 컴포넌트 일체** (플러그인, 서버 설정) — 서드파티 전제상 불가. 운영진 협업은 §9-D4의 선택적 확장으로만
- 애드온 서드파티 API 공개, 테마 스토어, 코스메틱, 멀티 서버 지원 — 트랙 B 이관
- 인증 아키텍처 변경 (v3에서 확정 — 재론 금지)

## 4. 시스템 아키텍처 (v3.1.0 목표 상태)

```
[동글랜드 런처 (Python + pywebview/WebView2)]
 ├─ 프론트: 단일 HTML (신규 디자인, CSS 변수 테마 구조)
 ├─ mll v8: 설치/검증/실행 (기존)
 ├─ MSA OAuth (기존) → UUID/토큰
 ├─ 로컬 IPC ────────────┐          ┌──> [자체 백엔드 (신규)]
 └─ API/WS 클라이언트 ───┼──────────┤     ├─ HTTPS API 서버
                          │          │     ├─ DB: profiles/friends/parties/news
                          ▼          │     └─ WebSocket (상태·파티 푸시)
[게임 클라이언트]                    │
 ├─ Fabric Loader + 필수 모드(기존)  │
 └─ 동글랜드 코어 모드 (신규) ───────┘ (직접 또는 런처 경유)
     ├─ 관찰 계층: 스코어보드/탭/액션바/채팅 파싱
     └─ HUD 렌더링 (파티, 타이쿤 정보)

[동글랜드 서버 — 제3자 운영, 통제 불가]
 └─ 관찰 대상일 뿐, 코드 배포·프로토콜 협의 전제 없음
```

신규 컴포넌트 2개: **자체 백엔드**, **코어 모드**. 소셜의 SSoT는 자체 백엔드이며, 게임 서버는 관여하지 않는다. 코어 모드↔백엔드 통신 경로는 §9-D3에서 결정 (모드 직접 연결 vs 런처 경유 로컬 IPC).

## 5. 기능 명세

### F1. 디자인 리뉴얼

| ID | 요구사항 |
|---|---|
| F1.1 | 사용자 제공 시안 HTML 기준으로 프론트 전면 교체. 기존 기능(계정, 설치/실행, 모드/셰이더 탭, 서버 상태, 업데이트 UX) 동등성 유지 |
| F1.2 | 시각 토큰을 CSS 변수(`--dl-*`)로 추출. 트랙 B 테마 시스템의 선행 구조 (v3.1.0에서는 내부 구조화까지, 테마 전환 UI 제외) |
| F1.3 | 단일 HTML 파일 구조 유지. 시안이 다중 파일이면 빌드 시 인라인 병합 |
| F1.4 | 렌더러 함수 공유 원칙 유지 (v3.0 회귀 방지) |
| F1.5 | (선택) pywebview 6 `window.state` 상태관리 리팩토링 — 시안 적용 공수와 함께 평가 |

수용 기준: 기존 E2E(로그인→설치→모드 선택→실행→자동접속) 전부 신규 UI에서 통과.

### F2. 자체 백엔드 최소 구축

| ID | 요구사항 |
|---|---|
| F2.1 | HTTPS API 서버 1대. 인증: MSA UUID + 세션 토큰 (로그인 시 Mojang 프로필 검증 후 발급 — 소유 증명 필수, UUID 사칭 방지) |
| F2.2 | DB: SQLite — `profiles`(uuid PK, 닉네임 캐시, last_seen), `friendships`(요청/수락 상태), `parties`(리더, 멤버), `news` |
| F2.3 | REST(최소): `POST /auth/session`, `GET /me`, `GET/POST /friends`, `POST /friends/requests/{uuid}/accept`, `GET/POST/DELETE /party`, `GET /news` |
| F2.4 | 실시간: WebSocket 1채널 — 접속 상태, 파티 초대/변경 푸시. 폴링 폴백 허용 |
| F2.5 | 게임 서버와 완전 독립 — 게임 서버 장애·변경이 백엔드에 영향 없음 (역도 성립) |

수용 기준: 런처 2대로 친구 요청→수락→상태 표시→파티 생성 왕복 확인.

### F3. 동글랜드 코어 모드 (클라이언트 사이드 전용)

**F3.1 개발 환경** — 동글랜드 서버 구동 버전에 종속 (§9-D1에서 실사)

| 서버 버전 | 툴체인 |
|---|---|
| 26.1+ | Java 25, Gradle 9.5.1, `net.fabricmc.fabric-loom` 1.17, Loader 0.19.3, 매핑 불필요 |
| 1.21.x | Java 21, 구 Loom(remap) + Mojang 공식 매핑, Yarn 미사용 (지원 종료) |

서버가 허용하는 클라이언트 버전이 복수라면 최신 안정 버전 1개만 타깃 (단일 버전 원칙).

**F3.2 기본 골격**

| ID | 요구사항 |
|---|---|
| F3.2.1 | 100% 클라이언트 사이드 모드 — 서버에 어떤 패킷도 추가 송신하지 않음 (기존 프로토콜 범위 내 동작). 서버 입장에서 바닐라 클라이언트와 구분점 최소화 |
| F3.2.2 | 기초 HUD 1개: 파티 멤버 표시 (F4 연동 검증용). v1 목표는 기능이 아니라 **런처↔백엔드↔모드 연동 파이프라인 검증** |
| F3.2.3 | 런처 필수 모드 파이프라인으로 배포 (기존 sha1 검증·launch blocking 재사용) |
| F3.2.4 | 모드 단독 실행(런처 없이 설치된 경우) 시에도 크래시 없이 비활성 상태로 동작 |

**F3.3 관찰 계층 (타이쿤 데이터 확보 — 서드파티의 핵심 기술)**

| ID | 요구사항 |
|---|---|
| F3.3.1 | 서버가 표준 프로토콜로 이미 보내는 정보를 구독: 스코어보드(사이드바), 탭리스트 헤더/푸터, 액션바, 보스바, 채팅, 타이틀 |
| F3.3.2 | 파싱 규칙을 하드코딩하지 않고 **정규식 기반 룰셋 파일**로 분리 — 서버가 표시 형식을 바꿔도 모드 재배포 없이 룰셋 업데이트(런처가 배포)로 대응 |
| F3.3.3 | 파싱 실패는 조용히 무시 (기능 비활성), 게임 진행에 영향 금지 |
| F3.3.4 | v1 범위: 룰셋 엔진 + 검증용 파싱 1종 (예: 스코어보드의 재화 표시)까지. 본격 타이쿤 HUD는 트랙 B |

수용 기준: 동글랜드 서버 접속 상태에서 스코어보드 파싱 결과가 HUD에 표시되고, 서버 표시 형식 임의 변경 시 크래시 없이 비활성화됨.

### F4. 소셜 기능 (자체 백엔드 완결형)

| ID | 요구사항 |
|---|---|
| F4.1 | 친구: UUID 기반 요청/수락/삭제. 닉네임은 Mojang API 캐시 |
| F4.2 | 접속 상태: online(런처 실행) / in-game(게임 프로세스 감지) / offline. 런처 하트비트 → 백엔드 → WebSocket 전파. **게임 서버 참여 없이 성립** |
| F4.3 | 파티: 생성/초대/수락/탈퇴, 리더 위임. 최대 인원 설정값 (기본 8) |
| F4.4 | 동반 접속: 리더 "함께 접속" → 멤버 런처에 확인 프롬프트 → 각자 quickPlayMultiplayer로 접속 (기존 자동접속 코드 재사용) |
| F4.5 | 인게임 파티 HUD: 데이터 경로는 백엔드→(런처 또는 직접)→코어 모드. §9-D3에서 결정 |
| F4.6 | 프라이버시: 상태 공개는 친구 한정, 차단 기능 포함 |

수용 기준: 2인 파티 → "함께 접속" → 두 클라이언트 서버 진입 + HUD 상호 표시.

## 6. 비기능 요구사항

| 분류 | 요구사항 |
|---|---|
| **서버 규칙 준수** | 코어 모드는 게임플레이 이점(자동화, 시야 확장, 자동 클릭 등) 기능을 포함하지 않음 — 정보 표시·소셜에 한정. 동글랜드 서버의 모드/클라이언트 규정을 확인하고 준수 (유저 제재 방지가 최우선) |
| 보안 | 백엔드 HTTPS 필수. 세션 토큰 저장은 기존 MSA 토큰 방식과 동일 수준. 관찰 계층 파싱은 방어적으로 (길이 제한, 예외 격리) |
| 안정성 | 백엔드 다운 시 런처 핵심 기능(로그인·설치·실행) 정상 — 소셜만 우아하게 비활성화 |
| 성능 | 소셜 WS·하트비트는 런처 프로세스에서만. 코어 모드는 이벤트 구동 파싱 + HUD 렌더 외 틱 로직 최소화 |
| 데이터 보호 | APP_DIR_NAME="DonglelandLauncher" 유지, 기존 설정 마이그레이션 규칙 준수 |

## 8. 테스트 계획 (요약)

1. **회귀**: 기존 E2E — 신규 UI에서 반복
2. **관찰 계층**: 실서버 접속 파싱 검증, 형식 변경 시뮬레이션(룰셋 불일치) 시 무해성, 바닐라 대비 성능 오버헤드 측정
3. **소셜**: 2계정 시나리오, 백엔드 다운 폴백, UUID 사칭 시도 거부 확인
4. **설치 신뢰성**: 필수 모드에 코어 모드 추가 후 sha1·launch blocking 확인
5. **규정 적합성**: 코어 모드 장착 상태로 서버 정상 플레이 장기 확인 (안티치트 오탐 관찰)

| D5 | 백엔드 호스팅 | 저가 VPS 권장 (게임 서버와 무관하므로 독립 호스팅 필연). WS 상시 연결 고려 |
| D6 | 릴리즈 분할 | §7 제안 승인 여부 |
